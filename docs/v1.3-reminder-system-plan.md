# V1.3 Automatic Booking Reminder System Plan

## Context & decisions

### Scheduling platform

- **Primary: Supabase Cron** — runs every 5 minutes.
- Supabase Cron can invoke an HTTP endpoint or a Postgres function. We use an HTTP call to a protected Astro endpoint.
- Vercel Hobby cron is not suitable because it runs once per day with low precision.

### Idempotency principle

The cron may run late, skip a beat, or overlap. The queue is processed with atomic `FOR UPDATE SKIP LOCKED` claims and a unique `(booking_id, notification_type, slot_version)` constraint so the same reminder can never be sent twice.

### Slot versioning

- Use the existing `audit_bookings.reschedule_count` as `slot_version`.
- Every reschedule increments `reschedule_count`, which naturally invalidates old reminders and creates new ones for the new slot.
- No new column is needed for slot versioning.

### Secret storage

- **Preferred:** `REMINDER_CRON_SECRET` is stored in **Supabase Vault** (`vault.secrets`).
- **Fallback:** read from `process.env.REMINDER_CRON_SECRET` if Vault is not enabled.
- The secret is never logged, never returned in API responses, and never sent to the client.

## 2. Data model

### New table: `public.booking_notifications`

```sql
CREATE TABLE public.booking_notifications (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),

  booking_id uuid NOT NULL
    REFERENCES public.audit_bookings(id)
    ON DELETE CASCADE,

  notification_type text NOT NULL
    CHECK (notification_type IN ('reminder_24h', 'reminder_1h')),

  slot_version int NOT NULL,

  scheduled_for timestamptz NOT NULL,

  status text NOT NULL DEFAULT 'pending'
    CHECK (status IN ('pending', 'processing', 'sent', 'failed', 'cancelled')),

  attempts int NOT NULL DEFAULT 0,
  next_attempt_at timestamptz,
  locked_at timestamptz,
  sent_at timestamptz,
  provider_message_id text,
  last_error text,

  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),

  UNIQUE (booking_id, notification_type, slot_version)
);

CREATE INDEX idx_booking_notifications_due
  ON public.booking_notifications (scheduled_for, status)
  WHERE status IN ('pending', 'failed');

CREATE INDEX idx_booking_notifications_booking
  ON public.booking_notifications (booking_id);
```

### Why `slot_version = reschedule_count`

- `reschedule_count` starts at 0 and increments on every successful reschedule.
- The unique constraint `(booking_id, notification_type, slot_version)` guarantees that a reschedule creates fresh reminders while old-slot reminders are cancelled, not duplicated.

## 3. Lifecycle integration

### Booking created

Hook after the booking is confirmed (calendar sync succeeded) in `src/pages/api/audit/book.ts`:

```ts
if (syncOutcome.overallStatus !== 'failed' && syncOutcome.overallStatus !== 'not_configured') {
  await scheduleBookingReminders(bookingId);
}
```

`scheduleBookingReminders(bookingId)`:

1. Fetch `selected_slot_start`, `reschedule_count` from `audit_bookings`.
2. Compute:
   - `reminder_24h.scheduled_for = slotStart - 24 hours`
   - `reminder_1h.scheduled_for = slotStart - 1 hour`
3. Skip any reminder whose `scheduled_for` is already in the past.
4. Insert both with `status = 'pending'`, `slot_version = reschedule_count`.
5. On conflict (unique constraint), do nothing.

Errors are logged and tracked but do not fail the booking response.

### Reschedule

Hook on successful reschedule in `src/lib/booking/rescheduleBooking.ts` (after step 15):

```ts
await rescheduleBookingReminders(bookingId);
```

`rescheduleBookingReminders(bookingId)`:

1. Cancel all `pending`/`failed` reminders for the booking.
2. Create new `reminder_24h` and `reminder_1h` for the new `selected_slot_start` and new `reschedule_count`.
3. Skip reminders whose `scheduled_for` is in the past.

### Cancel

Hook on successful cancel in `src/lib/booking/cancelBooking.ts`:

```ts
await cancelBookingReminders(bookingId);
```

`cancelBookingReminders(bookingId)`:

1. Update all non-sent reminders (`pending`, `processing`, `failed`) to `cancelled`.

### New file

- `src/lib/booking/reminderScheduling.ts` — contains `scheduleBookingReminders`, `rescheduleBookingReminders`, `cancelBookingReminders`.

## 4. Atomic queue processing

### RPC function: `claim_due_reminders`

A Postgres function performs atomic claiming so overlapping cron runs cannot process the same row.

```sql
CREATE OR REPLACE FUNCTION public.claim_due_reminders(
  batch_size int DEFAULT 20,
  processing_timeout interval DEFAULT '5 minutes'
)
RETURNS TABLE (
  id uuid,
  booking_id uuid,
  notification_type text,
  slot_version int,
  scheduled_for timestamptz,
  attempts int
)
LANGUAGE plpgsql
SECURITY INVOKER
SET search_path = pg_catalog, public
AS $$
BEGIN
  RETURN QUERY
  WITH due AS (
    SELECT bn.id
    FROM public.booking_notifications bn
    WHERE bn.status IN ('pending', 'failed')
      AND bn.scheduled_for <= now()
      AND (
        bn.status = 'pending'
        OR bn.next_attempt_at IS NULL
        OR bn.next_attempt_at <= now()
      )
      AND (
        bn.locked_at IS NULL
        OR bn.locked_at < now() - processing_timeout
      )
    ORDER BY bn.scheduled_for ASC
    LIMIT batch_size
    FOR UPDATE SKIP LOCKED
  )
  UPDATE public.booking_notifications bn
  SET
    status = 'processing',
    attempts = bn.attempts + 1,
    locked_at = now(),
    updated_at = now()
  FROM due
  WHERE bn.id = due.id
  RETURNING bn.id, bn.booking_id, bn.notification_type, bn.slot_version, bn.scheduled_for, bn.attempts;
END;
$$;

REVOKE EXECUTE ON FUNCTION public.claim_due_reminders(int, interval)
FROM PUBLIC, anon, authenticated;
```

### Stuck-record recovery

Records in `processing` longer than `processing_timeout` become claimable again because the claim query checks `locked_at < now() - processing_timeout`.

## 5. Pre-send verification

Before sending any reminder email, the worker re-reads `audit_bookings` and aborts if any of the following is true:

- `booking_status <> 'booked'`
- `calendar_sync_status <> 'synced'`
- `reschedule_count <> notification.slot_version`
- `selected_slot_start` changed
- `selected_slot_start` is in the past
- notification status is not `processing`
- notification status is `sent` or `cancelled`

If aborted, mark the notification as `cancelled` (or `failed` depending on reason) and log.

## 6. Retry rules

| Attempt | Result | Action |
|---------|--------|--------|
| 1–3 | Temporary failure | `status = 'failed'`, `next_attempt_at = now() + backoff` |
| 1–3 | Success | `status = 'sent'`, `sent_at = now()`, `provider_message_id = ...` |
| > max attempts | Failure | `status = 'failed'`, send admin alert |

Backoff suggestion:

- 1st retry: +5 minutes
- 2nd retry: +15 minutes
- 3rd retry: +30 minutes

Max attempts: **5**.

Admin alert on permanent failure:

- Send email to `env.adminEmail` with booking ID, notification type, slot, and last error.
- Track `reminder_permanently_failed` audit event.

The user-facing email must never be sent twice. This is enforced by:

- Unique constraint preventing duplicate reminder records.
- `processing` status preventing parallel sends.
- `sent` status preventing re-send.

## 8. Cron endpoint

### New file

- `src/pages/api/cron/audit-reminders.ts`

### Behaviour

- Only accepts `POST`.
- Returns `Cache-Control: no-store`.
- Reads the cron secret from:
  1. Supabase Vault (`vault.secrets` where name = `'reminder_cron_secret'`).
  2. `process.env.REMINDER_CRON_SECRET` fallback.
- Validates the `Authorization: Bearer <secret>` header with timing-safe comparison.
- On invalid/missing secret: returns `401` with no details.
- On success:
  1. Calls `claim_due_reminders(batch_size := 20)` via RPC.
  2. For each claimed notification:
     - Re-read booking.
     - Verify pre-send conditions.
     - Decrypt management token.
     - Send reminder email.
     - Update notification status.
  3. Returns JSON summary only:
     ```json
     { "claimed": 3, "sent": 2, "failed": 1, "skipped": 0 }
     ```
- Logs are structured and contain no personal data.

### Environment variable

```env
REMINDER_CRON_SECRET=<random 32+ char secret>
```

### Supabase Cron job

Created via SQL after deployment:

```sql
SELECT cron.schedule(
  'audit-reminders-every-5-minutes',
```

## 11. Post-review revisions

The following critical changes were applied after the initial plan review.

### 11.1 `claim_due_reminders` now handles three categories

- `pending` records whose `scheduled_for <= now()`;
- `failed` records whose `next_attempt_at <= now()` and `attempts < max_attempts`;
- stuck `processing` records whose `locked_at` has exceeded the timeout.

This ensures that temporary Resend failures respect the backoff and that worker crashes do not leave records stuck in `processing`.

### 11.2 `finalize_reminder_cancelled` RPC added

When the pre-send check decides not to send an email, the record is terminated into `cancelled` with a machine reason:

- `booking_not_booked`
- `calendar_not_synced`
- `stale_slot_version`
- `appointment_passed`

This prevents skipped records from being re-claimed indefinitely.

### 11.3 Automatic reminder repair / reconciliation

The cron endpoint calls `repairUpcomingReminders(48)` at the start of every run. It idempotently creates missing `reminder_24h` and `reminder_1h` records for upcoming `booked + synced` bookings using `ON CONFLICT DO NOTHING`. This recovers from transient scheduling failures without manual intervention.

### 11.4 Updated deployment order

1. Commit and push.
2. Generate `REMINDER_CRON_SECRET`.
3. Add `REMINDER_CRON_SECRET` to Vercel Production env **before** deploy.
4. Apply `007_booking_notifications.sql`.
5. Deploy the application.
6. Run `test-reminder-e2e.sh`.
7. Make one authenticated manual `POST` to `/api/cron/audit-reminders`.
8. Store `reminder_cron_secret` and `reminder_cron_url` in Supabase Vault.
9. Schedule the cron job.
10. Run the backfill.
11. Lifecycle smoke test.
12. Monitor 24–48 hours.

The DB migration is backward-compatible, so applying it before the code deploy is safe.

### 11.5 No env fallback for Supabase Cron

The Supabase Cron reads `reminder_cron_secret` and `reminder_cron_url` from Vault. The Astro endpoint reads `REMINDER_CRON_SECRET` from `process.env`. There is no env fallback on the cron side.

### 11.6 Admin alert is best-effort

The admin failure email uses an idempotency key (`audit-reminder-failure/{notification_id}`). `admin_alerted_at` is only updated when Resend accepts the send. If the admin alert itself fails, it is logged and tracked, but the notification is still finalized to `failed`.
